# Results

GPU sample detection (Gururani & Lerch 2017 reimplementation + adaptations).
Status: 2026-06-11, full pipeline including the random-forest stage.
Production config: log-frequency front end (367 bins, 4/semitone), K=32/L=20,
41 pitch shifts (0.25 st), 60 NMF iterations, TF32 + fast-math, both RTX
A5000s; RF = paper §3.4 (200 trees, √13 features), out-of-fold evaluation.

## Accuracy

### Headline comparison

| | Paper (with RF classifier) | Ours, before RF | Ours, after RF |
|---|---|---|---|
| Dataset | private, 80 sample/song pairs | **Sample100** (public), 103 relations | same |
| Candidate pool per query | 10 (1 true + 9 random) | **64** (all originals) | same |
| Decision type | binary (thresholded classifier) | ranking (heuristic score) | thresholded classifier prob |
| Song-level ("macro") precision | 83.3% | — (no threshold yet) | **50.0%** |
| Song-level recall | 50.0% | hit@1 **11.4%** / hit@3 **21.4%** (rank-based analog) | **100.0%** |
| Song-level F-measure | 62.5% | — | **66.7%** |
| MRR | not reported | **0.214** | **0.557** (hit@1 **50.0%**, hit@3 **57.1%**) |
| Location-level ("micro") P / R / F | 79.4 / 34.6 / 48.2% | — | — (per-occurrence eval not implemented) |

### Why the numbers are not directly comparable (read before citing)

1. **Pool size**: the paper detects against 10 candidates (random rank-1
   baseline 10%); we rank against 64 (random baseline 1.6%, random MRR ≈ 0.07).
   Our task is ~6× harder by chance alone.
2. **Dataset**: the paper's 80-pair corpus was never published (see CLAUDE.md
   log); Sample100 is the public benchmark and skews hard (chopped loops,
   short one-shots, heavy production).
3. **Decision type**: the paper's P/R/F come from a *trained random forest*
   making thresholded binary calls — the stage we have not added yet. Our
   pre-RF numbers are pure ranking by heuristic score; precision is undefined
   without a decision threshold. The "after RF" column is the
   apples-to-apples-in-form comparison.

### Pre-RF detail (full benchmark, 70 evaluable queries × 64 candidates)

- MRR **0.214** (≈ true original at effective rank ~5; random ≈ 0.07)
- hit@1 8/70 (11.4%), hit@3 15/70 (21.4%)
- Per-config frontier: `plots/sweep/pareto.png`, `plots/sweep/sweep_results.csv`
  (K=32 selected over K=40: MRR 0.214 vs 0.203, and faster; hop2048 traded
  −0.035 MRR for 2× speed — rejected as default)
- Controlled verification (exact ground truth): self-match, synthetic insert
  at +0.00 st, +6% resample at +1.01 st — all detected at the exact shift and
  location (`tests/run_ladder.sh`, docs/TESTING.md)

### RF stage detail (the headline "after RF" numbers)

- Corpus: 17.3M feature rows (70 queries × 64 candidates ×
  top-32-hypothesis path groups), 51,652 positives (0.30%), generated by
  `gpu_detect --features` in ~30 min on both GPUs.
- Evaluation is leakage-safe: 5-fold GroupKFold over the 52 connected
  components of the query↔original graph (queries sharing any original
  co-travel); all reported predictions are out-of-fold; the decision
  threshold (0.600, unanimous across folds) is chosen on each fold's TRAIN
  side; features are path geometry only (no heuristic scores, no metadata).
- Out-of-fold row AUC 0.629; per-fold 0.55–0.71.
- Reranking by max out-of-fold probability per (query, candidate):
  MRR 0.214 → **0.557**, hit@1 11.4% → **50.0%**, hit@3 21.4% → **57.1%**.
- Macro caveat, read honestly: recall 100% means every query's top candidate
  cleared the threshold — Sample100 has no sample-free queries, so the
  false-positive behavior on clean songs is unmeasurable and macro precision
  here equals top-1 accuracy. The paper's 50% recall reflects its
  conservative threshold on a task with rejectable pools.
- Feature importances confirm the paper's §3.3 thesis: path geometry
  dominates (avg_slope 0.130, avg_cost 0.107, min_cost 0.100, best_slope
  0.099); the pre-RF failure mode — melodic-resemblance confounds whose
  cheap-but-crooked alignments outrank true samples — is exactly what these
  separate. Training: `tools/train_rf.py`, model in `plots/rf/`.

## Performance

| Workload (warm caches, --iters 60, 2 GPUs) | Original | Now | Speedup |
|---|---:|---:|---:|
| 5-candidate full-song scan | 42.0 s | **3.0 s** | **14×** |
| Full-song query vs 64-candidate library | ~340 s | **19.3 s** | **18×** |
| 15 s clip vs 64-candidate library | 21.7 s | **2.3 s** | **9.4×** |
| Full 70-query benchmark | ~8.5 h (est.) | **~13 min** | **~40×** |

Per-iteration optimization history with gate evidence: README.md
("Optimization iterations" table) and the CLAUDE.md build log; per-iteration
profiles in `plots/00_baseline` … `plots/05_mel`; kernel-level analysis
(occupancy, stalls, tensor-core utilization, idle decomposition) in
`plots/PERF-CHARACTERIZATION.md`.

